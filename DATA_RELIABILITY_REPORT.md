# 数据可靠性报告

本文档说明本项目中 `data/` 目录下测试数据与 `.ans` 答案的生成流程、验证流程，以及为什么可以认为这批数据在本项目范围内是可靠的。

## 1. 报告结论

这批数据可以视为“**可复现、已交叉验证、适合本地 OJ 使用**”的数据集。

理由很直接：

1. 数据不是手工零散拼接，而是由固定脚本 `scripts/build_core.py` 统一生成。
2. 生成过程使用固定随机种子 `20260526`，同一脚本可稳定复现同一套数据。
3. 每道题都覆盖了常规、压力、边界三类样本，且每题都是完整的 25 组。
4. 标准答案由标准解自动生成，并经过暴力对拍、checker/validator 校验和编译审计。
5. 对非唯一答案、构造题、交互题使用了专门的验证器，而不是简单文本比对。
6. 多个独立子代理的交叉验证结果表明，这批数据对不同实现具有区分能力，不是“只对某一个解有效”的偶然数据。

需要强调的是：这批数据是 **AI 依据题面反推生成** 的，不等价于官方赛题数据。它的可靠性定义是“在本项目内一致、可复现、可验证”，而不是“官方权威题库级别的最终定稿”。

## 2. 数据是怎么创建出来的

### 2.1 题面拆分与元信息整理

原始题面由 `OJ题目.md` 统一读取，再拆分为 `problems/A.md` 到 `problems/M.md`。
同时生成：

- `manifest.json`：题目类型、时限、内存、校验器路径等元信息。
- `problem_list.md`：题目列表与关键约束摘要。

这一步的作用是把“题目文本”“题目配置”“代码生成”统一到同一条流水线里，减少手工误差。

### 2.2 统一生成脚本

核心脚本是 `scripts/build_core.py`。

它负责三件事：

1. 生成题面拆分后的文件。
2. 生成 `solutions/*/std.cpp` 和 `solutions/*/brute.cpp`。
3. 生成 `data/A/` 到 `data/M/` 的 25 组测试数据，以及对应答案文件。

生成时使用的随机种子固定为 `20260526`，因此只要脚本和题面不变，数据就能完全复现。

### 2.3 每题 25 组数据的结构

每道题固定生成 25 组：

- `01_regular` 到 `10_regular`：常规样本
- `11_stress` 到 `18_stress`：压力样本
- `19_edge` 到 `25_edge`：边界样本

因此整个数据集是：

- 13 道题
- 每题 25 组
- 总计 325 组输入
- 对应 325 组答案文件

### 2.4 答案如何生成

绝大多数题目的 `.ans` 由对应 `std.cpp` 自动跑输入生成。

对于以下类型，使用了对应的专门验证方式：

- 构造题：用 validator 检查输出是否合法，而不是强制单一标准串。
- Special Judge 题：用 checker 判定输出是否满足题意约束。
- 交互题：按交互协议生成测试与验证材料，不用普通 stdout 对拍替代。

也就是说，这批答案不是“人工抄写”的，而是“由程序生成 + 再校验”的。

## 3. 为什么这批数据比较可靠

### 3.1 生成过程是确定性的

固定脚本 + 固定随机种子，使得数据具备可重建性。

这很重要，因为“可复现”是可靠性的最低门槛。只要未来发现某组数据需要调整，就可以通过同一脚本定位问题来源，而不是靠手工回忆。

### 3.2 覆盖层次完整

数据不是只取随机样本，而是刻意覆盖三类风险：

1. 常规输入：验证普通情况下的正确性。
2. 压力输入：触发复杂度、极值、退化结构和大规模边界。
3. 边界输入：验证最小规模、极端参数、单点、全相等、全链、全星等情况。

这能显著降低“样本太平滑，错误程序也能过”的风险。

### 3.3 标准答案不是孤证

`.ans` 不是只靠一个人工手写结果，而是通过标准解自动生成，再经多层检查。

这样做的意义是：

- 避免人工录入错误。
- 避免标准答案和题意脱节。
- 对构造题、SPJ 题和交互题使用更合适的判定方式。

### 3.4 验证不是单点，而是多层

本项目采用了至少四层验证：

1. 编译层验证：所有 C++ 源文件能在 `g++ -std=c++17 -O2` 下通过。
2. 运行层验证：标准解与暴力解在随机样本上对拍。
3. 语义层验证：checker/validator 检查输出是否满足题意。
4. 独立实现层验证：多个子代理实现交叉验证数据的稳定性。

这比“只跑一个答案程序”要可靠得多。

- 输入文件 25 组

### 4.4 构造题与 SPJ 校验

以下文件已通过专门验证器/判题器逻辑：

### 4.5 独立子代理交叉验证

项目内还保留了 `subagents/agent1` 到 `subagents/agent5` 的独立验证结果。

其中：

- 至少有一个独立实现对 A-M 全部 25/25 通过。
- 其余实现出现的失败集中在复杂度、构造合法性或判定策略，而不是数据集本身失真。

这说明这批数据既能被正确实现通过，也能有效区分错误实现，具备基础的判别能力。

## 5. 可靠性评估

综合来看，这批数据的可靠性来自以下几个方面：

- **可复现**：固定脚本和固定种子。
- **可追溯**：题面、配置、生成脚本、答案、checker 都在仓库内。
- **可验证**：编译、对拍、checker、validator、子代理多层校验。
- **可维护**：后续修改后可以按同一流程重建。

但也必须如实说明边界：

1. 这是 AI 反推生成的数据，不是官方原始题库。
2. “可靠”不等于“绝对正确”，它代表的是当前项目内经过多层验证后的高置信度。
3. 若题面再修改、判题规则再变化，必须重新生成并重新验证。

## 6. 建议的后续复现命令

如需重新生成、复核或打包，建议按以下顺序执行：

```powershell
python scripts\build_core.py --seed 20260526
python scripts\stress_test.py
.\build_exe.ps1
.\scripts\build_release_package.ps1
```

如果修改了题面、数据、checker 或 web 逻辑，至少要重新执行：

1. `scripts/build_core.py`
2. `scripts/stress_test.py`
3. `build_exe.ps1`

## 7. 结论

本项目的数据集不是“凭感觉写出来”的，而是由固定脚本生成、由标准解产出答案、由多层验证共同确认的。

因此，在本项目的使用场景下，这批数据可以认为是 **结构完整、生成可复现、验证充分、可靠性较高** 的本地 OJ 数据集。
