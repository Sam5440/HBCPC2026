# M 题错误代码通过评测原因与补点报告

## 1. 结论

这份代码确实是错误的，当前旧数据没有覆盖它的核心漏洞，因此它能通过原有 `data/M/01` 到 `25`。

主要问题有两个：

1. 代码把“进入某只狗的视线格”当作一次性事件，但没有识别后续格子是否仍然属于同一只狗的视线。
2. 原 checker 只检查输出的狗是否可行，没有检查题面要求的“行坐标最小、再列坐标最小”。本次已同步补严 checker。

本次新增 `data/M/26_edge.in` 和 `data/M/27_edge.in`，专门覆盖“正确路径必须连续穿过同一只睡着狗的多个视线格”的场景。这类旧错误代码会输出 `-1 -1`，而正确答案是 `1 2`。

## 2. 错误代码的关键逻辑

代码预处理每个空地格子被哪些狗看到：

```cpp
vector<vector<vector<PII>>> dog(n + 1, vector<vector<PII>>(m + 1));
```

BFS 状态中记录：

```cpp
struct node {
    int x, y;
    int cnt;
    int dogx, dogy;
};
```

其中 `cnt` 表示路径上已经遇到过多少只狗的视线。移动到新格子时，代码使用：

```cpp
if (cnt + dog[nx][ny].size() > 1) continue;
```

如果新格子被一只狗看到，就直接：

```cpp
ncnt++;
ndogx = dog[nx][ny][0].first;
ndogy = dog[nx][ny][0].second;
```

这个模型是错误的。

正确含义应该是：选择一只狗睡着后，该狗的整条视线全部失效。因此路径可以穿过这只狗视线中的任意多个格子，而不是只能穿过一个格子。

错误代码没有判断：

```text
新格子看到的狗 == 已经选择睡着的狗
```

所以当路径需要连续走过同一只狗的两个视线格时，它会把第二个格子误判为“又需要睡一只狗”，从而错误拒绝。

## 3. 新增反例

新增数据 `26_edge`：

```text
1
4 3
.D#
S..
#.#
E..
```

坐标如下：

```text
(1,2) 是一只向下看的狗 D
(2,1) 是 S
(4,1) 是 E
```

狗 `(1,2)` 的视线覆盖：

```text
(2,2), (3,2), (4,2)
```

如果让 `(1,2)` 这只狗睡着，则可行路径为：

```text
(2,1) S
-> (2,2)
-> (3,2)
-> (4,2)
-> (4,1) E
```

其中 `(2,2)、(3,2)、(4,2)` 都是同一只狗的视线格。正确模型下这是合法的，因为只需要睡着 `(1,2)` 这一只狗。

错误代码执行过程：

1. 从 `S` 走到 `(2,2)`，发现该格被 `(1,2)` 看到，于是 `cnt = 1`。
2. 再从 `(2,2)` 走到 `(3,2)`，仍然是同一只狗 `(1,2)` 的视线。
3. 但代码计算 `cnt + dog[3][2].size() = 1 + 1 = 2`，于是直接拒绝。
4. 最终错误输出 `-1 -1`。

正确答案为：

```text
1 2
```

## 4. 为什么旧数据没有卡掉它

我用这份错误代码跑了原有 `data/M/01` 到 `25`，并使用 M checker 判定，结果全部通过。

原因是旧数据中没有构造出这种必要路径：

```text
必须连续穿过同一只睡着狗的多个视线格
```

已有数据要么：

- 不需要睡狗即可到达；
- 睡狗后只需要经过该狗视线中的一个关键格；
- 或者不存在方案。

因此错误代码虽然模型不正确，但没有在旧数据上暴露。

## 5. Checker 同步修复

题面要求如果有多种方案，需要输出字典序最小的狗坐标：

```text
优先行坐标最小；若行坐标相同，优先列坐标最小。
```

原 `checker/M/checker.cpp` 对非 `-1 -1` 输出只做了：

```cpp
if (!can_sleep(grid, x, y)) return 1;
```

这会接受任意一个可行狗，而不是只接受最小坐标的可行狗。

本次修复为先枚举所有狗，找到第一个可行狗：

```cpp
pair<int,int> best = {-1, -1};
for (auto [r, c] : dogs) {
    if (can_sleep(grid, r, c)) {
        best = {r, c};
        break;
    }
}
```

然后要求输出必须等于 `best`：

```cpp
if (x == -1 && y == -1) {
    if (best.first != -1) return 1;
} else {
    --x; --y;
    if (best != make_pair(x, y)) return 1;
}
```

由于扫描顺序是行优先、列优先，这正好对应题面要求的字典序最小坐标。

## 6. set 版本继续分析

后续给出的 `set<PII>` 版本已经修掉了第 2 节所述的重复计数问题：它会把路径上经过的狗视线集合去重，因此连续走过同一只狗的多个视线格不会再被误判为两只狗。

它仍然有一个高风险剪枝：

```cpp
vector<vector<PII>> vis(n + 1, vector<PII>(m + 1, {n + 1, m + 1}));
```

同一个位置只保留一个字典序最小的狗坐标。也就是说，如果某个位置可以通过两条路径到达：

```text
路径 A：已经选择了坐标更小的狗
路径 B：已经选择了坐标更大的狗
```

代码会直接丢弃路径 B，即使路径 A 后续无法到达终点，而路径 B 后续可以到达终点。

这个思路指出的是正确的风险点：状态不应该只按位置压缩，理论上还需要区分“当前已经选择睡着哪只狗”。否则局部最小狗不一定能推出全局最优。

不过，用户给出的两个参考样例在当前标准解和 `set` 版本上都会输出相同答案：

```text
28_edge -> 1 3
29_edge -> 1 4
```

因此它们适合作为边界补点保存，但不能单独卡掉这个 `set` 版本。

针对该剪枝漏洞，新增 `data/M/31_edge.in`：

```text
1
6 7
#D##D##
S....E#
.#.#..L
.#...##
.###.##
.....##
```

正确答案是：

```text
1 5
```

其中 `(1,2)` 是字典序更小的狗，但睡它后无法到达终点；睡 `(1,5)` 可以走通。错误 `set<PII>` 版本会先用 `(1,2)` 到达并标记若干汇合格，随后把带着 `(1,5)` 到达这些格子的状态剪掉，最终输出：

```text
-1 -1
```
